## 3. Зафиксированная архитектура (не переоткрывать)

**Единица — суб-флот** = (тип × компоновка × класс дальности) из Cirium. Отдельный борт = только параметры (кресла, дальность, крейсерская), НЕ ось прогноза. Прогноз «борт → маршрут → месяц» запрещён.

**Архитектура — top-down, якорь на парк:**
1. `рейсы ≈ N × availability × active_intensity × дни × сезонность`
2. Раскладка по сети (доли, помесячно).
3. Детерминированная конверсия.

**Конверсия (рейсы — единственная свободная величина):**
- `циклы = рейсы`
- `км = рейсы × длина_сектора` (геометрическая константа)
- `блок_часы = рейсы × блок_тайм`, `блок_тайм ≈ фикс_накладка + расстояние / крейсерская`
- `пасспоток = кресла × LF`
- Когерентны по построению.

**Якорь разнят на два наблюдаемых множителя:**
- `active_intensity` = интенсивность на **летавший** борт (циклы/сутки или блок-часы/сутки), жёсткий физ. потолок.
- `availability` = (бортов с ≥1 вылетом) / (бортов в строю по Cirium) за месяц. Ловит короткий простой.
- Долгое хранение → через `status=stored` в Cirium (эффективный размер парка), НЕ через availability (иначе двойной счёт).

**Ведущая метрика по плечу (binding):** короткое → **циклы/сутки**; длинное → **блок-часы/сутки**. Cirium-пределы (дальность, суточный налёт) — **фильтр-предохранитель** поверх. Дальность как драйвер отдельно НЕ считаем (впечатана в наблюдаемую сеть).

**Прогноз вперёд = репликация типового года.** Уровень — с пост-разгонного плато (последние ~12 мес), плоско. Разгон 2022→2024 НЕ протягиваем. Тренд — только подписанный сценарий.

**Архетип — из поведения + Cirium, ТРИ ортогональные оси, разные источники** (уточнено, см. §4 и таблицу ниже):

| Ось | Что | Источник | Статус |
|---|---|---|---|
| Конфигурация борта | пакс / карго / VIP | Cirium `[Primary Usage]` | закрыт |
| Гранулярность/физика | тип, кресла, дальность, крейсерская | Cirium `[Aircraft Sub Series]` | закрыт |
| Регулярность сети | расписание ↔ on-demand | поведение, PC1 | закрыт вердиктом |
| Характер контракта / ACMI-режим | своя сеть / ACMI-донор / -заказчик | **нет источника** | подписанное ограничение |

**Критично по атрибуции полей (правка узла 21):** Cirium — классификатор *железа*, не операции. `[Primary Usage]` даёт назначение **планера** (Passenger/Cargo/Business), НЕ характер перелёта. ACMI-режим не наблюдаем **ни в одном** источнике: Cirium его не знает (это не железо), поведение читает ACMI-донора как регуляр (он летает по чужому расписанию). Прежняя формулировка «ACMI из Cirium» была неверна — ACMI это неотделимое ограничение, а не решённый узел.
- `[Primary Usage] = Cargo` → нет пассажиропотока, конверсия на тонно-км/загрузку.
- `[Primary Usage] = Business` → низкая плотность кресел; но фракцион-vs-принципал даёт PC1, не это поле.
- `[Aircraft Sub Series]` → ключ группировки суб-флота + физические константы.

**Панель ~6-8 млн = иерархический ПРИОР, НЕ календарный наклон.** Даёт: уровни утилизации, сезонную форму, потолки архетипа, регуляризацию тонких ячеек. НЕ даёт форвард-тренд (в панели тот же пост-COVID разгон). Ось обмениваемости: **архетип → класс плеча → тип ВС → возрастная корзина**. Усадка адаптивна по объёму ячейки. Вклад перевозчика балансируется.

**Возраст:** страта в пулинге (интерсепт availability). БЕЗ наклона деградации, БЕЗ выбытия по возрасту.

**N вперёд = снимок + подтверждённые приходы Cirium.** Новый борт **сразу зрелый**. Прибавочные рейсы **уплотняют** замороженную сеть по текущим долям. **Уходов в источнике нет** → N монотонно не убывает → смещение вверх (подписано).

**Доли сети — помесячные, сезонная маска (СИЛЬНАЯ версия).** Сезонный маршрут **зануляется** вне сезона. Карта декабря ≠ карта июля. Набор пар заморожен, но сезонная маска включения/выключения поверх него живёт. Сезонная форма тонких маршрутов пулится по архетипу.

## 4. Вердикт шага 1 — РЕЗУЛЬТАТ (это факт, на нём строим дальше)

Прогон на 6,143,719 рейсах, 623 ячейки (431 complete-case в PCA), референс-метки присутствуют.

- **PC1 = 60.7% дисперсии, все четыре сигнатуры грузятся положительно** → это ось регулярности, как задумано. Порядок референс-меток по PC1 совпадает с ожидаемым (Spearman 0.77).
- **PC1 — ДИСКРЕТНЫЕ моды.** 2 моды KDE, bimodality coeff 0.68 (порог ~0.56). **Выживает** фильтр экспозиции (на чистых ячейках 2 моды, BC 0.72) и **выживает** огрубление гранулярности → on-demand кластер реален, не артефакт тонких ячеек. Впадина (антимода) на **PC1 ≈ −0.30**: on-demand мода = 167 ячеек (70% low-conf), schedule мода = 264 ячейки (7% low-conf).
- **Лакмус фракциона отработал отрицательно.** Ожидали 2D (регулярность × интенсивность). Фракцион НЕ отщепился по PC2 — он на on-demand-конце **PC1** (сепарация 2.77 по PC1 против 0.40 по PC2). Пространство для решения об архетипе **одномерно**. Проще, чем ждали.
- **PC2 (22%) реален, но это НЕ вторая ось архетипа.** Им целиком правит S2 (нагрузка +0.92). Это **ось недельного ритма** (день-недельная периодичность), а не on-demand-измерение. Раскол расписание↔on-demand — на PC1.
- **S2 немонотонна по регулярности.** Борт, летающий одни волны 7 дней/нед, имеет НИЗКУЮ вариацию по дню недели → низкий S2 → выглядит нерегулярным (обратный порядок: LCC 0.05 < FSC 0.12 < regional 0.27). **S2 меряет недельную ФОРМУ, не регулярность.** → S2 переезжает из оси архетипа в **сезонный член якоря**. Решение об архетипе де-факто стоит на **трёх** сигнатурах (S1, S3, S4).
- **ACMI читается как регулярный** (charter_acmi mean PC1 = +0.86, ждали ~0.25). Сигнатуры не сломались: долгосрочный wet-lease летает по расписанию чужого перевозчика. **Подтверждает** узел 21 (характер контракта из поведения не достаётся) — не дефект.
- **Карго недопредставлен** (1 из 4 ячеек в complete-case) → scheduled-cargo угол пока не валидирован, провизорно.

**Следствия для даунстрима (зафиксированы вердиктом):**
- **Якорь (шаг 2):** PC1 как дискретные бакеты, рез по антимоде (~−0.30). Бакет-специфичная тугость якоря. **On-demand бакет требует своего intensity-модели** (low-recurrence, demand-constrained — N×утилизация там кратно завышает).
- **Пулинг (шаг 3):** пул внутри бакета; PC2 несём как вторую координату пулинга для сезонного/недельного разложения.

> Вердикт **порогово-вычислен**. Первичное доказательство — KDE/scatter PNG. Подтвердить глазами до лока: смотри впадину −0.30 и что on-demand облако не сливается с расписанием на хвосте.

## 5. ПЕРВАЯ ЗАДАЧА: тонкий вертикальный срез до LOCO

Цель — не полный движок, а **минимальный сквозной прогноз, достаточный чтобы прогнать LOCO и прочитать покрытие.** Это гейт по премиссе.

### 5.1 Два блокера входа (дёшево, закрыть ДО среза)
1. **Перегенерировать панель** на полной истории Cirium (июль-2023, месячные снапшоты). Вердикт §4 считался на выгрузке с *аппроксимированной* атрибуцией оператора; ядро (дискретность PC1) устоит, но цифры освежить на чистых данных.
   - Заодно проверить **327 хвостов без даты поставки**: это борта с вводом *до* окна (тогда дата некритична, летали всё окно) или пропуск записи для вошедших *внутри* окна (тогда смещение вверх)? Померить долю вторых.
2. **Частотка `[Primary Usage]`** по массиву на NULL/`Multiple`. Заметная доля → fallback-логика (назначение из плотности кресел `Sub Series`) до того, как конверсия начнёт молча ветвиться и часть парка выпадет.

### 5.2 Что строим в срезе (огрублённо, только schedule-бакет)
Берём **один тестовый перевозчик из schedule-бакета** (well-behaved большинство, 264 ячейки). On-demand-форк в срез НЕ включаем — атакуем отдельным заходом, не смешивая ошибку ядра с ошибкой форка.

- **N(t):** снимок парка + подтверждённые приходы Cirium, помесячно.
- **availability_sf(месяц)** = (хвостов с ≥1 вылетом) / (хвостов in-service по Cirium), оценка на трейлинг-плато. Пул по архетипу — в срезе можно грубо (среднее бакета).
- **active_intensity_sf** = циклы/сутки или блок-часы/сутки на **летавший** борт, уровень плато. Binding-метрика по классу плеча.
- **Якорь:** `рейсы_мес ≈ Σ_sf [ N_sf(t) × availability_sf × active_intensity_sf × дни × сезонность_sf(месяц) ]`
- **Сеть (грубо):** наблюдаемые помесячные доли, замороженный набор пар + сезонная маска. Полные маски НЕ строим — грубого разложения достаточно для гейта.
- **Конверсия:** циклы/км/часы детерминированно; пасспоток только если `Primary Usage=Passenger` (иначе тоннаж/пропуск).

### 5.3 Что заморожено до прохождения среза (НЕ строить)
- Продакшн-проектор архетипа (LDA/логит на метках) — для среза unsupervised-рез PC1 по антимоде достаточен.
- Полные сезонные маски сети.
- Тонкость иерархического пулинга.
- **On-demand intensity-форк.**

Все они — фиделити поверх фундамента, чья валидность ещё не подтверждена. Достраивать сейчас = вкладывать в опцион, который LOCO может обнулить.

### 5.4 LOCO — сам гейт
- Прячем тестового перевозчика **целиком** из панели.
- Гоним движок на панельных приорах + его собственная история до отсечки.
- Прогноз вперёд на held-out окне, **где парк/сеть не менялись** (чтобы не спутать промах движка с нарушением посылки в реальности).
- Сравнение **покомпонентно** (active_intensity, сезонная форма, доли сети) И по агрегату четырёх метрик. Один чёрно-ящичный итог по сумме рейсов почти бесполезен.
- **Reliability-кривая:** 80%-полоса накрывает факт в ~80%? Плот номинального покрытия против эмпирического.
- **Чтение:** ложится на диагональ → премисса держит. Систематически узка → премисса где-то расходится с миром → возвращаемся пересматривать **допущение**, не реализацию.

Атрибуция теперь чистая (июль-2023) → промах LOCO **диагностичен по модели**, не по данным. Это повышает ценность среза как гейта. (Раньше пришлось бы инструментировать долю рейсов с неоднозначным оператором — теперь не нужно.)

## 6. Порядок сборки ПОСЛЕ прохождения среза

Только если LOCO на срезе показал приемлемое покрытие:
1. **On-demand intensity-форк** (demand-constrained модель для второго бакета).
2. **Продакшн-проектор архетипа** на референс-метках.
3. **Полный пулинг** (иерархия страт, адаптивная усадка, балансировка оператора, вывод параметрической полосы).
4. **Полные сезонные маски** сети.
5. **Полный LOCO + послойный бэктест** на замороженном снимке, reliability по горизонту.

Дисциплина: не наращивать фиделити слоёв, пока премисса не прошла первую встречу с реальностью на срезе.
